General Overview of Avathon Renewable Suite API's to access the data

Avathon APM API Interface

Avathon Asset Performance Management (APM) Platform provides flexibility for users to visualize, query and report using the SCADA and Alarms data. In some cases users may want to write their own programs on top of the data stored by Avathon. To do so we make it easy by providing users access to the data via REST API's

What is REST API?

A REST API (also known as RESTful API) is an application programming interface (API or web API) that conforms to the constraints of REST architectural style and allows for interaction with RESTful web services. REST stands for representational state transfer.

An API is a set of definitions and protocols for building and integrating application software. It’s sometimes referred to as a contract between an information provider and an information user—establishing the content required from the consumer (the call) and the content required by the producer (the response). For example, when using Avathon's REST API the caller or consumer will call APM API's to get certain information and as an information provider APM API will return a JSON response with that information.

In other words, if you want to interact with a computer or system to retrieve information or perform a function, an API helps you communicate what you want to that system so it can understand and fulfill the request.

You can think of an API as a mediator between the users or clients and the resources or web services they want to get. It’s also a way for an organization to share resources and information while maintaining security, control, and authentication—determining who gets access to what.

Handling Pagination in External APIs

When interacting with External APIs, it’s crucial to handle pagination effectively to ensure you retrieve all available data. The API response will typically include query and data fields. The query field contains metadata, including number_of_pages, while the data field holds the actual results.

To retrieve the complete dataset, you should iterate through all pages as specified by number_of_pages in the query field. Keep in mind that the first page might sometimes be empty, while subsequent pages contain data. Therefore, even if the initial request returns no data, continue to fetch the remaining pages until all specified pages have been processed. This approach guarantees that no data is missed, providing you with a comprehensive collection of the API's resources.

API Gateway Rate, Burst, and Quota Limits

Rate Limits:

Rate limits define the maximum number of API requests a client can make within a specified time period. These limits help to ensure the stability and availability of the API by preventing overuse by any single client. Rate limits are typically expressed in terms such as requests per second (RPS).

Burst Limits:

Burst limits allow short-term spikes in traffic beyond the rate limit. This is useful for handling sudden increases in demand without immediately throttling the client. Burst limits are designed to accommodate brief periods of higher-than-normal traffic. For example, if the rate limit is 10 RPS, the burst limit might allow up to 20 RPS for a short duration before enforcing the rate limit again.

Quota Limits:

Quota limits control the total number of API requests a client can make over a longer period, such as a day or month. These limits are useful for managing overall API consumption and ensuring fair usage among all clients. For instance, a quota limit might allow a client to make 50,000 requests per day.
